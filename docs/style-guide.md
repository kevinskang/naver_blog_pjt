# 프로젝트 스타일 가이드

목표: 가독성 유지, 일관성 확보, 협업 용이성 향상

## 네이밍

- 모듈/패키지: 소문자와 밑줄 (`example_module.py`)
- 클래스: CapWords (`MyClass`)
- 함수/변수: snake_case (`do_action`)

## 타입 힌트

- 공개 함수·메서드는 가능한 한 타입 힌트를 사용
- 복잡한 타입은 `typing` 또는 `typing_extensions`로 별도 정의

## 문서화

- 모든 public 함수/클래스에 docstring (Google 스타일 또는 numpydoc)
- 모듈 상단에 간단 설명 추가

## 포맷팅

- `black` 기본 설정 (줄 길이 88)
- import 정렬은 `isort` (프로젝트 규칙에 맞춰 구성)

## 린트/검사

- `ruff`로 빠른 정적 검사, `mypy`로 타입 검사 일부 적용
- 불필요한 복잡도(중복, 긴 함수)는 리팩토링 권장

## 함수 크기 & 책임

- 함수는 한 가지 책임에 집중. 80라인 초과 함수는 분해 고려
- 모듈은 명확한 책임 경계를 가지도록 분리

## 예외 처리

- 커스텀 예외는 `utils/exceptions.py`에 정의
- 에러 핸들링은 별도 레이어 (`error_handler.py`)로 중앙화

## 테스트

- 단위 테스트는 핵심 로직(유틸, 재시도, 예외 매핑)에 우선 적용
- 통합 테스트는 자동화 흐름(로그인, 업로드, 글쓰기) 검증

## 간단 체크리스트

- PR 전: `ruff`, `black --check`, `mypy`(선택) 실행
- 변경 후: 관련 테스트 추가 및 기존 테스트 통과 확인

참고: `pyproject.toml`의 도구 설정과 일치하도록 유지
